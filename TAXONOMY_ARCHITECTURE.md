# Taxonomy Architecture

## Authority

Tags are event-level taxonomy signals. Timelines inherit tags only through their events.

Taxonomy records must be normalized, searchable, and relationship-backed. A tag record without event usage is allowed for administration, but public discovery value exists only when the tag is linked through `event_tags`.

## Scope

This document governs:

- `tags`
- `event_tags`
- tag governance metadata
- tag aliases, redirects, and merges
- tag pages
- tag search behavior
- timeline relatedness derived from tags
- tag cleanup and normalization

## Data Model

Core tables:

```text
tags(id, slug, name)
event_tags(event_id, tag_id)
tag_governance(tag_id, ...)
tag_aliases(tag_id, alias, alias_slug)
tag_redirects(source_slug, target_tag_id)
tag_merges(source_tag_id, target_tag_id, source_slug)
```

`event_tags` is the source of truth for event-to-tag relationships.

## Ownership

Backend-owned:

- Tag creation and update.
- Tag validation.
- Event-to-tag relationship persistence.
- Taxonomy governance reporting.
- Public tag detail resolution.

Frontend-owned:

- Displaying resolved tags.
- Submitting selected tag IDs through validated admin APIs.

The frontend must not infer tag relationships or access the database directly.

## Tag Creation Rules

Tags may be created by:

- Import execution.
- Relationship backfill.
- Admin tag management.
- Seed scripts in development.

Rules:

- Slug is canonical for identity.
- Slug must be generated by the shared slugification behavior.
- Name must be human-readable.
- Duplicate slug creation must reuse the existing tag.
- Import and backfill paths must deduplicate tags by slug before persistence.

## Event Relationship Rules

Events own taxonomy context through `event_tags`.

Required invariant:

```text
event has tag context -> event_tags row exists
```

Timeline tags must be derived from distinct event tags, not stored independently on timelines.

## Public Behavior

Tag pages:

- Resolve the tag by slug.
- Return timelines by joining `tags -> event_tags -> timeline_events -> timelines`.
- A tag with no `event_tags` rows produces an empty tag page.

Search:

- Timeline search ranks tag names and slugs through `event_tags`.
- Milestone search ranks tag names and slugs through `event_tags`.

Related timelines:

- Related timeline logic uses tags derived from event relationships.
- Missing `event_tags` disables tag-based relatedness.

SEO:

- Timeline and milestone JSON-LD use resolved tag names.
- Empty event tag relationships produce weak or missing keyword metadata.

## Composite Tag Policy

Composite tags are invalid when they represent delimiter-separated atomic concepts, for example:

```text
Rome;Politics;Military
```

Correct representation:

```text
Rome
Politics
Military
```

Composite tags must not be created by future imports. Existing composite tags must be handled through a separate zero-loss normalization process after event relationships have been recovered.

## Normalization Order

Correct remediation order:

1. Recover missing `event_tags` from source CSVs.
2. Validate tag usage through public pages, search, and JSON-LD.
3. Normalize composite tags into atomic tags.
4. Preserve existing public URLs until redirect policy is approved.
5. Clean up deprecated or orphaned tags only in a separate migration.

Do not normalize tags before recovering relationships; production tag records alone do not prove event ownership.

## Governance

Taxonomy governance is diagnostic and editorial. It should surface:

- orphaned tags
- duplicate candidates
- unreviewed tags
- promotion candidates
- deprecated tags
- merge and redirect history

Governance metadata must not silently change public behavior unless explicitly designed.

## Validation Checklist

Before taxonomy-changing releases:

- `event_tags` count is non-zero in production.
- Public tag pages return expected timelines for sampled tags.
- Timeline detail pages show expected tags.
- Milestone detail pages show expected tags.
- Search returns expected results for sampled tags.
- Sitemap does not amplify unusable empty tag pages without an explicit SEO decision.

## Change Control

Any taxonomy behavior change requires:

- Migration plan for existing data.
- Rollback plan.
- Search impact review.
- SEO impact review.
- Tests or dry-run reports for tag parsing and relationship counts.
