# Permanent Context V2

## Status
Status: Active starter permanent context

Authority Level: Highest long-term project context authority

Version: 2.0 starter

## Purpose
This document preserves long-lived truths, platform memory, institutional boundaries, and accepted decisions that future work must inherit.

## Scope
This document governs:
- Stable project context.
- Locked platform baselines.
- Institutional memory.
- Accepted wave outcomes.
- Long-term boundary decisions.
- Known capability maturity.

Out of scope:
- Temporary task notes.
- Unaccepted proposals.
- Implementation details that belong in architecture or schema canon.

## Authority
Permanent context should be used to orient future programs before implementation begins. It does not override product, architecture, schema, experience, or execution canon in their direct domains.

## Ownership
Owner: Institutional memory steward

Contributors:
- Product authority steward
- Architecture authority steward
- Governance authority steward when established

## Principles
- Preserve accepted decisions.
- Distinguish locked capabilities from planned capabilities.
- Keep institutional boundaries explicit.
- Record what is true now, not what might become true later.
- Avoid turning roadmap ideas into authority before acceptance.

## Locked Baseline
- Chronology is locked.
- Historical Authority is locked.
- Historical Context is locked.
- Timeline View -> Milestone -> Historical Context -> Historical Object is authoritative.
- Historical Objects remain secondary surfaces.
- Participation is the public context unit.
- Governance is planned, not operationally complete.
- Historical Library is architected, not operationally implemented.
- Factory is partial.

## Institutional Boundaries
- Factory owns Production Memory.
- Historical Library owns Published Memory.
- Historical Object Registry owns authority identity.
- Platform presents approved public read models.
- Editors and Automation require explicit governance boundaries before operational use.

## Change Management
Permanent context changes must:
- Cite the accepted program or audit that changed project truth.
- Avoid speculative wording.
- Preserve historical decisions even when future programs supersede implementation details.
- Update references when canon documents become authoritative.
